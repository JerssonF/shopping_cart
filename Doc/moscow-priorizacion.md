# Priorización sencilla (MoSCoW)

Este documento organiza las tareas del proyecto según su importancia, para decidir qué hacer primero y qué dejar para después.

¿Qué significa cada grupo?

- Must Have: cosas que el proyecto necesita sí o sí para funcionar.
- Should Have: cosas importantes que pueden esperar si hace falta tiempo.
- Could Have: mejoras útiles si queda tiempo.
- Won't Have: cosas que no haremos ahora.

Must Have (imprescindible)

- Un servicio que guarde y gestione carritos.
- Formas básicas para añadir, ver, cambiar y quitar productos del carrito.
- Mostrar el total del carrito.
- Guardar los datos para que no se pierdan.
- Una forma sencilla para que la página o la aplicación hable con el servicio.
- Una página básica para que el usuario vea y gestione su carrito.

Should Have (importante pero no urgente)

- Revisiones para evitar errores en los datos que envía el usuario.
- Manejo claro de errores para que el usuario entienda qué pasó.
- Mejor presentación de las rutas entre la página y el servicio.
- Guardado más detallado de la información si es necesario.
- Mejoras visuales en la página.
- Más instrucciones para ejecutar todo junto en un entorno de pruebas.

Could Have (deseable si hay tiempo)

- Una interfaz más atractiva y adaptable a móviles.
- Opciones de búsqueda o filtros dentro del carrito.
- Un historial simple de acciones del carrito.
- Pruebas automáticas básicas para detectar problemas.

Won't Have (no en esta etapa)

- Registro y control de usuarios avanzado.
- Conexión con pasarelas de pago.
- Despliegue en servidores externos como paso inicial.
- Arquitectura con muchos servicios separados.

Conclusión

Con esta lista buscamos centrarnos primero en lo esencial para tener una versión simple y usable. Las mejoras se añadirán después según el tiempo y las prioridades.
